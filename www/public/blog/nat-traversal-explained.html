Understanding NAT Traversal and Why P2P Connections Work

NAT traversal is the technique that makes direct browser-to-browser file transfer possible even when both devices sit behind firewalls and routers. Tools like 4share use NAT traversal protocols — STUN, TURN, and ICE — to discover public network addresses and punch holes through NAT devices, establishing direct encrypted connections between two browsers without routing file data through any server. Without NAT traversal, most peer-to-peer connections would fail before they could even begin.

This article explains the different types of NAT, why they block P2P connections, and exactly how the protocols used by WebRTC overcome these obstacles to enable direct file transfer.

What Is NAT and Why Does It Exist?

NAT stands for Network Address Translation. It is a technique used by routers and firewalls to translate between private (internal) IP addresses and public (external) IP addresses. NAT exists because of a fundamental limitation of the internet: there are not enough public IPv4 addresses for every device.

How NAT Works

When your device connects to a router, the router assigns it a private IP address (like 192.168.1.100) that is only valid within your local network. When your device needs to reach the internet, the router translates the private address to its own public IP address. To the outside world, all devices on your network appear to share a single IP address.

NAT Diagram (text representation):

  Private Network                    Public Internet
  ┌──────────────┐
  │ Device A     │──┐
  │ 192.168.1.10 │  │
  └──────────────┘  │    ┌─────────────┐     ┌──────────────┐
                     ├───►│   Router    │────►│   Internet   │
  ┌──────────────┐  │    │  NAT Layer  │     │              │
  │ Device B     │──┘    │ 203.0.113.5 │     │              │
  │ 192.168.1.20 │       └─────────────┘     └──────────────┘
  └──────────────┘
  
  Router maps: 192.168.1.10:54321 ──► 203.0.113.5:40001
               192.168.1.20:54321 ──► 203.0.113.5:40002

The router maintains a translation table that maps each outgoing connection to a unique port on its public IP. When responses arrive, the router consults this table and forwards them to the correct internal device.

Why NAT Is a Problem for P2P

NAT was designed for a client-server model: your device initiates outbound connections to servers, and the router handles the responses. This works perfectly for web browsing, email, and streaming. But P2P connections require something different — one peer needs to initiate a connection to another peer, but the other peer's address is a private IP that is unreachable from the internet.

When Device A behind NAT tries to connect to Device B behind a different NAT, the connection attempt arrives at Device B's router. But Device B's router has no mapping for this incoming connection — it was never initiated from inside the network. The router drops the packet. The connection fails.

Types of NAT: Full Cone, Restricted Cone, Port Restricted, and Symmetric

Not all NAT devices behave the same way. The type of NAT determines how restrictive it is and, consequently, how difficult it is to establish a P2P connection through it.

Full Cone NAT

Full Cone NAT is the least restrictive type. Once an internal device sends a packet to an external address, the NAT creates a mapping. Any external device can then send packets back to that mapping, and they will be forwarded to the internal device.

Internal device sends to ANY external address
  ──► NAT creates mapping: internal:port ──► external:port
  ──► ANY external device can send to external:port
  ──► NAT forwards to internal device

Full Cone NAT is P2P-friendly. Once one peer sends a packet outward (creating a mapping), the other peer can send directly to that mapping. No special traversal techniques are needed beyond discovering the public address.

Restricted Cone NAT

Restricted Cone NAT is slightly more restrictive. After the internal device sends a packet outward, the NAT creates a mapping, but only packets from the specific IP address that received the initial packet are forwarded. Packets from other IP addresses are dropped.

Internal device sends to External Device X
  ──► NAT creates mapping
  ──► Only External Device X can send back
  ──► Packets from External Device Y are DROPPED

For P2P, this means the second peer must first receive a packet from the first peer before it can send to the first peer. This requires careful choreography during hole-punching.

Port Restricted Cone NAT

Port Restricted Cone NAT adds another layer: not only must the incoming packet come from the correct IP address, it must also come from the correct port. This is the most common type of NAT found in home routers.

Internal device sends to External IP:Port
  ──► NAT creates mapping for that specific IP AND port
  ──► Only packets from that exact IP:Port are forwarded
  ──► Packets from same IP but different port are DROPPED

Port Restricted Cone NAT makes hole-punching trickier because both peers must send their initial packets to each other's exact public address and port simultaneously.

Symmetric NAT

Symmetric NAT is the most restrictive. Unlike the other types, it creates a different mapping for each destination. If Device A sends a packet to Server X and Server Y, the NAT assigns different external ports for each destination.

Internal device sends to Server X: port 80
  ──► NAT assigns external port 40001

Internal device sends to Server Y: port 80
  ──► NAT assigns external port 40002 (DIFFERENT port!)

This makes P2P connections extremely difficult. When Device A sends a packet through NAT to Device B, the NAT assigns a port. But Device B does not know which port was assigned (it is different from what the STUN server reported), so it cannot send directly back to Device A.

The Problem: Why NAT Breaks P2P Connections

The fundamental problem is this: NAT was designed to allow outbound connections but block unsolicited inbound connections. In a client-server model, this is fine — the client initiates the connection, and the NAT allows the response back in. But in a P2P model, both devices need to initiate connections to each other, and neither can reach the other's private address.

Consider two devices behind NAT:

Device A (192.168.1.10)          Device B (10.0.0.15)
       │                                │
       ▼                                ▼
  Router A                         Router B
  203.0.113.5                     198.51.100.3
       │                                │
       └────────── Internet ────────────┘

Device A wants to connect to Device B.
But Device B's private address (10.0.0.15) is unreachable from Device A's network.
And Device A's private address (192.168.1.10) is unreachable from Device B's network.

Device A cannot send a packet directly to Device B because 10.0.0.15 is a private address that does not exist on the public internet. Device A can only send packets to public addresses. Similarly, Device B cannot reach Device A's private address.

Even if both devices know each other's public addresses (obtained via STUN), the routers may still block the incoming connections because there is no existing mapping for them.

NAT Hole-Punching: The Core Technique

NAT hole-punching is the technique that makes most P2P connections possible. The basic idea is simple: both peers send outgoing packets to each other simultaneously (or near-simultaneously), creating NAT mappings that allow the other peer's packets to pass through.

How Hole-Punching Works

  1. Peer A sends a packet to Peer B's public address. This creates a mapping on Router A: internal:port → external:port. Router A now knows that packets from Peer B's public address should be forwarded to this mapping.
  2. Peer B sends a packet to Peer A's public address. This creates a mapping on Router B.
  3. The packets cross in transit. When Peer A's packet arrives at Router B, Router B checks its mapping table. If Peer B has already sent a packet to Peer A's address (creating the mapping), Router B forwards the packet to Peer B. The same happens in reverse at Router A.
  4. Connection established. Both routers now have mappings for the other peer's packets. Direct communication can proceed.

Hole-Punching Diagram (text representation):

Time 1:  Device A ──► Router A ──► Device B's public address
         (creates mapping on Router A)

Time 2:  Device B ──► Router B ──► Device A's public address
         (creates mapping on Router B)

Time 3:  Router A checks mapping ──► forwards to Device A ✓
         Router B checks mapping ──► forwards to Device B ✓

Result:  Device A ◄══════════════════════════► Device B
         Direct, encrypted P2P connection established!

Why Timing Matters

The timing of the hole-punching packets matters because NAT mappings are not permanent. They typically expire after a few minutes of inactivity. Both peers must send their initial packets close together (within a few seconds) so that both NAT mappings are active when the other peer's packet arrives.

If there is a large delay — for example, Peer A sends its packet, but Peer B takes 30 seconds to respond — Router A's mapping may have expired by the time Peer B's packet arrives. The packet is dropped, and the hole-punching attempt fails.

This is why WebRTC's ICE agent sends multiple candidates and tries them in rapid succession. The more candidates and the faster the attempts, the higher the chance that at least one pair of NAT mappings aligns correctly.

STUN: Discovering Your Public Address

Before hole-punching can work, each peer must know its own public IP address and port mapping. This is where STUN comes in.

What STUN Does

STUN (Session Traversal Utilities for NAT) is a protocol defined in RFC 8489. A STUN server is a lightweight server that runs on the public internet. When your browser sends a STUN binding request, the server responds with the source address it saw — which is your public IP address and port mapping.

Your Device (192.168.1.10:54321)
       │
       │ STUN Binding Request
       ▼
STUN Server (public internet)
       │
       │ STUN Binding Response:
       │ "I received your packet from 203.0.113.5:40001"
       ▼
Your Device learns: "My public address is 203.0.113.5:40001"

The STUN server does not relay data. It simply reports back the address from which the request arrived. This is enough for the peer to construct a public address that the other peer can try to connect to.

STUN Limitations

STUN works well for most NAT types, but it has limitations:

  • Symmetric NAT: When Device A contacts the STUN server, it learns one port mapping. But when Device A tries to send to Device B, the NAT assigns a different port. The port learned from STUN is not the port that Device B will see.
  • Port-restricted NAT with strict behavior: Some NATs only allow traffic from the exact IP:port that the initial packet was sent to. STUN gives you the mapping, but the other peer must also send a packet first before the mapping accepts incoming data.
  • UDP blocking: Some corporate firewalls block all UDP traffic. Since STUN and hole-punching use UDP, they cannot function when UDP is blocked.

For these cases, TURN is needed as a fallback.

TURN: The Relay Fallback

TURN (Traversal Using Relays around NAT) is a protocol that provides a relay server for cases where direct P2P connection is impossible. When STUN and hole-punching fail, TURN steps in to relay all data between the two peers.

How TURN Works

A TURN server is a public server that allocates a relay address for a client. The client sends data to the relay address, and the TURN server forwards it to the destination. Both peers connect to the same TURN server, which shuttles data between them.

Device A                    TURN Server                   Device B
    │                           │                            │
    │──── Allocate Relay ───────►│                            │
    │◄─── Relay Address ────────│                            │
    │                           │                            │
    │──── Send Data ────────────►│──── Forward Data ─────────►│
    │◄─── Receive Data ─────────│◄─── Receive Data ──────────│
    │                           │                            │

TURN adds latency (data must travel to the relay and then to the destination, instead of going directly) and costs bandwidth (the relay server must handle all data for the connection). But it ensures connectivity even in the most restrictive network environments.

When TURN Is Needed

  • Both peers behind symmetric NAT: Hole-punching cannot work because the port assignments are unpredictable.
  • UDP blocked by firewall: TURN can operate over TCP (port 443), which is rarely blocked.
  • Corporate security policies: Some organizations block all P2P traffic. TURN relays through port 443, which is typically allowed.
  • Mobile carrier networks: Carrier-grade NAT often behaves like symmetric NAT, making direct connection unreliable.

TURN Performance

TURN relay introduces overhead because data takes an indirect path. In practice, TURN-relayed connections are about 50–70% as fast as direct connections, depending on the TURN server's location and load. For file transfers, this means longer transfer times, but the transfer still works.

ICE: Putting It All Together

ICE (Interactive Connectivity Establishment) is the framework that orchestrates NAT traversal in WebRTC. It combines STUN, TURN, and hole-punching into a systematic process for finding the best available connection path.

The ICE Process

  1. Gather candidates: Each peer collects all possible connection addresses:
    • Host candidates: local network interfaces (private IPs)
    • Server-reflexive candidates: public addresses discovered via STUN
    • Relay candidates: relay addresses allocated via TURN
  2. Exchange candidates: Both peers send their candidate lists to each other through the signaling server.
  3. Pair candidates: ICE creates pairs of candidates (one from each peer) and sorts them by priority (local first, then STUN, then TURN).
  4. Check pairs: ICE sends connectivity checks (STUN binding requests) on each pair to see if the path works.
  5. Select best pair: The first pair that successfully passes connectivity checks is selected. ICE nominates it as the active connection.
  6. Establish connection: The DTLS handshake runs over the selected pair, and the data channel opens for file transfer.

ICE Candidate Priority:

Priority order (highest to lowest):

1. Host candidate ◄── Direct LAN connection (fastest, most reliable)
2. Server-reflexive (STUN) ◄── NAT traversal (works for most NAT types)
3. Relay (TURN) ◄── Fallback relay (slowest, but always works)

ICE tries candidates from highest to lowest priority. If a host candidate works (both peers on the same LAN), it is used — no NAT traversal needed. If not, ICE tries STUN candidates. Only if STUN fails does ICE fall back to TURN.

The Symmetric NAT Challenge

Symmetric NAT is the hardest NAT type for P2P traversal. Here is why it is so problematic:

The Problem with Symmetric NAT

When Device A behind symmetric NAT sends a packet to STUN server X, the NAT assigns port 40001. When Device A sends a packet to STUN server Y, the NAT assigns port 40002. Each destination gets a different port.

For hole-punching to work, Device B needs to send a packet to the exact port that Device A's NAT assigned for the path to Device B. But neither Device A nor Device B knows this port in advance — it is assigned by the NAT only when the first packet is sent to that specific destination.

Device A ──► STUN Server X ──► NAT assigns port 40001
Device A ──► Device B ────────► NAT assigns port 40003 (DIFFERENT!)
Device B ──► Device A (port 40001) ──► NAT DROPS (wrong port!)

Solutions for Symmetric NAT

When both peers are behind symmetric NAT, the only reliable solution is TURN relay. However, there are also some advanced techniques:

  • Port prediction: Some symmetric NATs assign ports sequentially (40001, 40002, 40003...). If the peer can predict the next port assignment, hole-punching may succeed. However, this is unreliable and does not work with all NAT implementations.
  • UPnP/PCP: Some routers support UPnP or PCP (Port Control Protocol), which allows applications to create port mappings automatically. If both routers support this, direct connection is possible regardless of NAT type. However, many routers disable UPnP for security reasons.
  • TURN relay: The reliable fallback. Always works, but adds latency and bandwidth cost.

Carrier-Grade NAT (CGNAT) on Mobile Networks

Mobile networks use a special form of NAT called Carrier-Grade NAT (CGNAT) or Large-Scale NAT (LSN). This is essentially symmetric NAT deployed at the carrier level, where thousands of mobile devices share a pool of public IP addresses.

How CGNAT Affects P2P

When you use mobile data, your phone's traffic goes through the carrier's CGNAT gateway. The carrier assigns a public port for each connection, but this port may change frequently and unpredictably. Some carriers reassign ports every few minutes, which can break existing P2P connections.

However, most modern mobile carriers use a form of port-restricted cone NAT rather than strict symmetric NAT. This means STUN-based hole-punching works for the majority of mobile connections. WebRTC's ICE agent handles this automatically by probing multiple candidates quickly.

WiFi vs. Mobile for P2P

  • WiFi (home network): Usually port-restricted cone NAT. STUN works well. High success rate for direct connection.
  • Mobile data: Usually CGNAT (port-restricted or symmetric). STUN works in most cases. TURN fallback more common than on WiFi.
  • Corporate WiFi: Often strict firewall with symmetric NAT. TURN relay is frequently needed.
  • Public WiFi (cafe, airport): Usually port-restricted cone NAT. STUN works, but bandwidth may be limited.

These numbers show that NAT traversal succeeds in the majority of cases, but a meaningful percentage of connections require TURN relay. This is why production P2P applications must include TURN infrastructure — it is not optional for reliable connectivity.

Factors That Improve Success Rates

  • Multiple STUN servers: Using STUN servers in different regions increases the chance of discovering a usable public address.
  • IPv6 support: IPv6 eliminates NAT entirely. When both peers support IPv6, direct connection is almost always possible. WebRTC can attempt IPv6 connections alongside IPv4.
  • UPnP/PCP: If the router supports automatic port mapping, the application can create a mapping that allows incoming connections, effectively turning a cone NAT into a full cone NAT.
  • TCP fallback: Some restrictive networks block UDP but allow TCP. WebRTC can fall back to TCP-based transport (TURN over TCP) for these environments.

Frequently Asked Questions

What is NAT and why does it block P2P connections?

NAT (Network Address Translation) is a technique used by routers to share a single public IP address among multiple private devices. NAT translates private IP addresses to the public address, and it blocks unsolicited incoming connections, which prevents P2P peers from reaching each other directly.

What is NAT traversal?

NAT traversal is the process of establishing a direct connection between two peers that are both behind NAT devices. It uses techniques like STUN hole-punching, TURN relaying, and ICE candidate gathering to find a working path between the two devices.

What is the difference between STUN and TURN?

STUN discovers your public IP address so peers can connect directly. TURN acts as a relay server when direct connection fails — data passes through the TURN server instead of going directly between peers.

How does ICE work in WebRTC?

ICE (Interactive Connectivity Establishment) gathers all possible connection paths between two peers — local, STUN-discovered, and TURN relay — and tries them in order. The first path that works is used for the connection.

What types of NAT exist and which are hardest to traverse?

The four main types are Full Cone, Restricted Cone, Port Restricted Cone, and Symmetric NAT. Symmetric NAT is the hardest to traverse because it assigns a different public port for each destination, making hole-punching unreliable.

Why do some P2P connections fail?

P2P connections can fail when both peers are behind symmetric NATs, when UDP is blocked by firewalls, or when corporate security policies restrict peer-to-peer traffic. TURN relay fallback handles these cases.

Can NAT traversal work on mobile networks?

Yes. Mobile networks use carrier-grade NAT (CGNAT), which is typically port-restricted cone NAT. WebRTC handles this using STUN and, if needed, TURN relay fallback.

Does 4share use NAT traversal?

Yes. 4share uses WebRTC with ICE for NAT traversal. It uses STUN servers to discover public addresses and falls back to TURN relay when direct connection is not possible.
