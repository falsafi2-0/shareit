WebRTC file transfer lets two browsers send files directly to each other without uploading anything to a server, and it is the core technology behind tools like 4share. By creating an encrypted peer-to-peer data channel between the sender's browser and the receiver's browser, WebRTC eliminates cloud storage, eliminates upload wait times, and keeps your data on your devices until the transfer is complete. This article explains exactly how WebRTC makes that possible, from the initial signaling handshake through to the encrypted byte stream that carries your file.

If you have ever used a browser-based video call, you have already used WebRTC. But the same technology that powers video calls can also carry arbitrary data, including large files. Understanding how it works under the hood reveals why peer-to-peer file transfer is faster, more private, and more reliable than traditional cloud-based methods.

What WebRTC Actually Is (and What It Is Not)

WebRTC stands for Web Real-Time Communication. It is an open standard maintained by the W3C and IETF that allows web browsers to establish direct peer-to-peer connections. The standard was originally designed for real-time audio and video communication, but its data channel capability makes it equally suited for file transfer.

WebRTC is not a library you install. It is a set of browser APIs — RTCPeerConnection, RTCDataChannel, and RTCSessionDescription — that are built into every modern browser. You do not need plugins, extensions, or native applications. JavaScript running in a web page has full access to WebRTC functionality.

The Three Pillars of WebRTC

WebRTC is built on three core components:

  • RTCPeerConnection: Manages the connection between two peers. Handles ICE candidate gathering, DTLS handshake, and media/data transport negotiation.
  • RTCDataChannel: The bidirectional data pipe that carries file data between peers. Supports ordered or unordered delivery, reliable or unreliable transport.
  • RTCSessionDescription (SDP): A data structure that describes the capabilities each peer supports — codecs, encryption methods, data channel parameters. Peers exchange SDP offers and answers to negotiate the connection.

For file transfer, the RTCDataChannel is the star. It provides a reliable, ordered, encrypted byte stream between two browsers. Think of it as a TCP socket, but one that runs directly between browsers without traversing any server.

Why WebRTC Was Chosen for File Transfer

Several properties make WebRTC ideal for browser-based file transfer:

  • Built-in encryption: DTLS is mandatory in WebRTC. You cannot create an unencrypted data channel.
  • NAT traversal: WebRTC includes ICE (Interactive Connectivity Establishment), which handles the hardest part of peer-to-peer networking — getting through firewalls and NAT devices.
  • Cross-platform: Works on Chrome, Firefox, Safari, and Edge across Windows, macOS, Linux, iOS, and Android.
  • No installation: Runs entirely in the browser. No app store, no downloads, no plugins.
  • Sub-millisecond latency: Data channels have minimal overhead, enabling high-throughput file transfers.

The Architecture of a WebRTC File Transfer

A WebRTC file transfer involves three participants: the sender's browser, the receiver's browser, and a lightweight signaling server. Here is how they interact:

│   Sender's   │◄──── SDP Offer / ICE ────►│         │◄──── SDP Answer / ICE ────►│   Receiver's  │
│   Browser    │         (WebSocket)        │  (Cloudflare  │         (WebSocket)        │   Browser     │
│              │                           │   Worker)     │                           │               │
│  File Data   │◄═══════════════════════════╪═══════════════╪═════════════════════════►│  File Data    │
│              │      Encrypted SCTP        │  (No server   │      Encrypted SCTP        │               │
└──────────────┘      Data Channel          │   involved)   └──────────────              └──────────────┘
                                             └───────────────┘

The signaling server's role is limited to the initial handshake. It helps the two browsers exchange their SDP offers/answers and ICE candidates. Once the direct peer-to-peer connection is established, file data flows directly between the two browsers. The signaling server never sees a single byte of file content.

Step-by-Step Connection Flow

  1. Sender creates an offer: The sender's browser generates an SDP offer describing its capabilities (supported codecs, data channel parameters, encryption methods).
  2. Offer is sent via signaling server: The SDP offer travels from the sender's browser to the signaling server, which forwards it to the receiver's browser.
  3. Receiver creates an answer: The receiver's browser generates an SDP answer accepting the connection parameters.
  4. Answer travels back: The SDP answer goes through the signaling server back to the sender.
  5. ICE candidates are exchanged: Both browsers gather ICE candidates — potential network paths to reach each other (local IPs, public IPs, TURN relay addresses). These are also exchanged via the signaling server.
  6. DTLS handshake: The browsers perform a DTLS handshake to establish encryption keys. This happens directly between the peers, not through the server.
  7. Data channel opens: An RTCDataChannel is ready for file data. The signaling server is no longer needed.
  8. File transfer begins: The sender reads the file, chunks it, and streams the chunks through the encrypted data channel to the receiver.

Signaling: The Matchmaker That Steps Aside

WebRTC requires a signaling mechanism to bootstrap the connection. This is the one place where a server is involved, and it is important to understand exactly what the server does — and does not do.

What the Signaling Server Handles

  • SDP exchange: Relaying the SDP offer from sender to receiver and the SDP answer from receiver to sender.
  • ICE candidate exchange: Relaying network path information between the two peers.
  • Room/session management: Matching a sender with the correct receiver using a transfer code or session ID.
  • Presence detection: Notifying each peer when the other is online and ready.

What the Signaling Server Never Does

  • It never sees file content. File data bypasses the server entirely.
  • It never stores files. No file data is ever written to the server's disk.
  • It never relays data. Once the direct connection is established, the server is out of the loop.
  • It never decrypts anything. The server has no access to the DTLS keys.

In 4share, the signaling server runs on Cloudflare Workers — a serverless platform that executes the signaling logic at the edge, minimizing latency for the initial handshake. The signaling messages are tiny (a few kilobytes) and ephemeral (deleted immediately after relay).

WebSocket-Based Signaling

Most WebRTC implementations use WebSockets for signaling because they provide a persistent, low-latency connection between the browser and the signaling server. Here is a simplified example of what a signaling message looks like:

// Sender creates an offer
const pc = new RTCPeerConnection({
  iceServers: [{ urls: 'stun:stun.cloudflare.com:3478' }]
});

const dc = pc.createDataChannel('filetransfer', {
  ordered: true,
  maxRetransmits: 3
});

const offer = await pc.createOffer();
await pc.setLocalDescription(offer);

// Send the SDP offer to the signaling server
signalingSocket.send(JSON.stringify({
  type: 'offer',
  sdp: pc.localDescription
}));

The message is a simple JSON object containing the SDP description. The signaling server forwards it to the receiver, which responds with an answer. This exchange typically completes in under 100 milliseconds.

ICE Candidates and NAT Traversal

The hardest part of peer-to-peer networking is not the encryption or the data transfer — it is getting through NAT devices and firewalls. Most devices sit behind a NAT (Network Address Translation) router that hides their private IP address from the outside world. WebRTC handles this through ICE.

How ICE Works

ICE (Interactive Connectivity Establishment) is a framework that discovers all possible paths between two peers and attempts them in order of preference:

  1. Host candidates: Each peer gathers its local network interfaces — the private IP addresses of the device. If both peers are on the same network, a direct connection is possible using these addresses.
  2. Server-reflexive candidates (STUN): If the peers are on different networks, each peer contacts a STUN server to discover its public IP address and port mapping. This is the address that the other peer can reach from the internet.
  3. Relay candidates (TURN): If direct connection and STUN both fail (symmetric NAT, restrictive firewall), ICE falls back to a TURN relay server. The TURN server acts as a proxy, forwarding data between the two peers.

ICE pairs up candidates from both sides and tries each combination. The first combination that successfully connects wins. This process is fast — typically completing in under 500 milliseconds.

Why STUN Is Usually Enough

Studies show that approximately 85–90% of WebRTC connections succeed using only STUN (no TURN relay needed). This is because most home and mobile NAT devices use cone NAT or port-restricted NAT, which are compatible with STUN-based hole-punching. Only about 10–15% of connections require a TURN relay, typically in corporate or highly restricted network environments.

The ICE Candidate Exchange

Both peers send their ICE candidates to each other through the signaling server. The candidates look like this:

// ICE candidate message
{
  "type": "candidate",
  "candidate": "candidate:1 1 udp 2122260223 192.168.1.100 54321 typ host",
  "sdpMid": "0",
  "sdpMLineIndex": 0
}

The candidate describes the transport protocol (UDP), the IP address, the port, and the type (host, srflx, or relay). Both peers run their ICE agent, which attempts connections using each candidate pair until one succeeds.

SCTP Data Channels: The Actual File Pipe

Once ICE and DTLS have established a secure connection, the RTCDataChannel carries file data over SCTP (Stream Control Transmission Protocol). SCTP runs on top of DTLS, which runs on top of UDP, creating a reliable, ordered, encrypted transport for file data.

Why SCTP Instead of TCP

You might wonder why WebRTC does not just use TCP. There are several reasons:

  • Head-of-line blocking: TCP is strictly ordered. If one packet is lost, all subsequent packets are stalled until the lost packet is retransmitted. SCTP supports partial reliability, meaning a file transfer can continue even if some packets are lost (as long as the application-layer protocol handles it).
  • Multistreaming: SCTP supports multiple independent streams within a single connection. A file transfer tool could use one stream for file data and another for control messages (progress updates, acknowledgments) without them interfering with each other.
  • NAT traversal: SCTP runs over UDP (via DTLS), which is better for NAT traversal than running over TCP. TCP-based NAT traversal often fails with symmetric NATs.

Data Channel Configuration for File Transfer

When creating a data channel for file transfer, you configure it for reliability and ordering:

const dc = pc.createDataChannel('filetransfer', {
  ordered: true,         // Packets arrive in order
  maxRetransmits: 3,     // Retransmit lost packets up to 3 times
  maxPacketLifeTime: 5000 // Or give up after 5 seconds
});

For file transfer, ordered: true is essential because file data must arrive in sequence. The maxRetransmits parameter limits how many times a lost packet is retried, preventing infinite retransmission stalls.

Flow Control and Backpressure

SCTP provides built-in flow control. If the receiver's buffer fills up because it cannot process data fast enough, SCTP automatically slows down the sender. This prevents buffer overflows and memory exhaustion. The sender's browser detects this backpressure through the bufferedAmount property on the data channel:

function sendChunk(chunk) {
  if (dc.bufferedAmount > MAX_BUFFER_SIZE) {
    // Wait for the buffer to drain before sending more
    setTimeout(() => sendChunk(chunk), 10);
    return;
  }
  dc.send(chunk);
}

This flow control mechanism is critical for large file transfers. Without it, the sender could overwhelm the receiver's memory or network buffer, causing the transfer to fail.

DTLS Encryption: Security Built Into the Protocol

WebRTC mandates encryption. Unlike HTTPS, where you can technically create an unencrypted HTTP connection, WebRTC requires every data channel to be encrypted using DTLS (Datagram Transport Layer Security). This is not optional — the browser will not create an unencrypted data channel.

How DTLS Works in WebRTC

After ICE selects a working candidate pair, the two peers perform a DTLS handshake. This handshake establishes shared encryption keys without either peer sending those keys over the network. The result is that all subsequent data channel traffic is encrypted using AES (Advanced Encryption Standard) with 128-bit or 256-bit keys.

The DTLS handshake happens automatically — the JavaScript API does not expose the keys or require any configuration. Security is guaranteed by the protocol, not by developer discipline.

End-to-End Encryption

Because the DTLS handshake happens directly between the two peers (not through the signaling server), the encryption is truly end-to-end. The signaling server that facilitated the initial handshake never has access to the encryption keys. Even if the signaling server were compromised, it could not decrypt the file data.

This is a significant advantage over cloud-based file transfer, where the cloud provider necessarily has access to your data (unless you encrypt it before uploading, which adds complexity and overhead).

File Chunking and Flow Control

WebRTC data channels have a maximum message size that varies by browser. Chrome typically supports messages up to 256 KB, Firefox up to 256 KB, and Safari up to 64 KB. To transfer files larger than these limits, file transfer tools split the file into smaller chunks and send them sequentially.

The Chunking Process

Here is how a typical WebRTC file transfer chunks and sends a file:

  1. Read the file: The sender uses the File API to read the file as an ArrayBuffer.
  2. Split into chunks: The ArrayBuffer is sliced into chunks of 64 KB to 256 KB, depending on browser support.
  3. Send each chunk: Each chunk is sent through the RTCDataChannel. The sender monitors bufferedAmount to avoid overwhelming the receiver.
  4. Receiver reassembles: The receiver collects chunks in order and writes them to a Blob or ArrayBuffer.
  5. Transfer complete: Once all chunks are received, the receiver assembles the complete file and offers it for download.

Chunk Size Optimization

The optimal chunk size depends on several factors:

  • Network MTU: The Maximum Transmission Unit is typically 1500 bytes for Ethernet. Chunks larger than the MTU get fragmented, adding overhead. However, SCTP handles fragmentation internally, so chunks can be larger than the MTU.
  • Browser memory limits: Larger chunks consume more memory on both sides. Very large chunks (over 1 MB) can cause memory pressure, especially on mobile devices.
  • Throughput: Larger chunks mean fewer API calls and less JavaScript overhead. But too large and you risk overwhelming the buffer.

For most applications, a chunk size of 64 KB to 256 KB strikes a good balance between throughput and memory usage. Some implementations dynamically adjust chunk size based on network conditions.

Metadata Message Format

Before the file data begins streaming, the sender typically sends a metadata message containing the file name, size, MIME type, and other properties. This allows the receiver to display accurate progress information and name the downloaded file correctly.

// Metadata message (sent before file data)
{
  "type": "metadata",
  "fileName": "project-archive.zip",
  "fileSize": 5368709120,
  "mimeType": "application/zip",
  "totalChunks": 81920,
  "chunkSize": 65536
}

Performance: Real-World WebRTC Transfer Speeds

WebRTC data channel performance depends on the network path between the two peers. Here are real-world benchmarks from different network scenarios:

Several factors influence these numbers:

  • Upload speed of the sender: This is often the bottleneck. Many home internet connections have asymmetric speeds, with upload bandwidth significantly lower than download bandwidth.
  • Download speed of the receiver: The receiver must be able to absorb data as fast as the sender pushes it.
  • Network latency: Higher latency reduces throughput due to SCTP's congestion control algorithm. A connection with 200ms RTT will have lower throughput than one with 20ms RTT, even on the same bandwidth.
  • Congestion: If other devices on the network are consuming bandwidth, the available throughput for the file transfer decreases.

Throughput Optimization Techniques

WebRTC file transfer tools use several techniques to maximize throughput:

  • Parallel data channels: Some implementations open multiple data channels to utilize available bandwidth more effectively.
  • Adaptive chunk sizing: Dynamically adjusting chunk size based on observed throughput and buffer levels.
  • Congestion control tuning: WebRTC's built-in congestion control (GCC — Google Congestion Control) adapts to network conditions, but can be tuned for bulk transfer scenarios.
  • Buffer management: Careful management of send and receive buffers prevents stalls and maintains high throughput.

Fallback Strategies When Direct Connection Fails

While WebRTC succeeds in establishing a direct connection in 85–90% of cases, there are scenarios where direct connection is impossible. Both peers behind symmetric NATs, corporate firewalls that block UDP, or strict security policies can prevent direct connectivity.

TURN Relay Fallback

When direct connection fails, ICE falls back to TURN (Traversal Using Relays around NAT). A TURN server acts as a middleman, receiving data from one peer and forwarding it to the other. This adds latency and reduces throughput, but ensures the transfer can proceed.

TURN relays are typically run by the same infrastructure that provides the signaling server. For 4share, Cloudflare's TURN servers provide global coverage with low latency.

Data Channel Limits and Workarounds

When TURN is not available or too slow, some implementations use alternative transport methods:

  • WebSocket relay: A WebSocket connection through the signaling server can relay file data when direct connection fails. This is slower than TURN but requires no additional infrastructure.
  • HTTP chunked transfer: Some tools fall back to chunked HTTP POST/GET through the server for very large files where data channel reliability is insufficient.

The key advantage of WebRTC is that data flows directly between peers. With WebSocket or HTTP, all file data passes through the server, which creates a bandwidth bottleneck, a privacy concern, and an infrastructure cost. For a P2P file transfer tool, WebRTC is the clear winner.

Why Not Just Use WebSocket?

WebSocket is simpler to implement and works in more network environments (no NAT traversal needed). But it has fundamental limitations for file transfer:

  • Server bandwidth costs: Every byte of file data passes through the server. For a 50 GB file, that is 50 GB of server bandwidth in AND 50 GB out — 100 GB total per transfer.
  • Privacy: The server operator can see all file data in transit. Even with WSS encryption, the server decrypts and re-encrypts the data.
  • Scalability: The server must handle all concurrent transfers. As usage grows, server costs grow linearly.

WebRTC avoids all three problems. The signaling server handles only tiny JSON messages. File data never touches any server. And the server infrastructure cost remains constant regardless of transfer volume.

Frequently Asked Questions

What is WebRTC and how does it enable file transfer?

WebRTC (Web Real-Time Communication) is a browser standard that allows direct peer-to-peer connections between two devices. For file transfer, it creates an encrypted data channel between the sender's browser and the receiver's browser, allowing files to flow directly without any intermediate server.

Is WebRTC file transfer secure?

Yes. WebRTC mandates DTLS (Datagram Transport Layer Security) encryption for all data channels. Every byte of file data is encrypted end-to-end between the two browsers. No intermediary can read the content.

How fast is WebRTC file transfer?

Speed depends on the network between the two peers. On a local network, WebRTC can saturate a gigabit Ethernet connection. Over the internet, speed is limited by the slower of the two peers' upload or download bandwidth.

Do both users need to be online at the same time?

Yes. WebRTC creates a live, synchronous connection between two browsers. Both peers must be online and have their browser open for the transfer to occur.

Can WebRTC transfer files between mobile and desktop?

Yes. WebRTC works across all modern browsers on both mobile and desktop platforms, including Chrome, Firefox, Safari, and Edge. A phone can send files directly to a laptop and vice versa.

What file size limits does WebRTC have?

WebRTC itself has no file size limit. The practical limits are determined by browser memory and the availability of chunked streaming implementations. Modern WebRTC file transfer tools can handle files of any size, including multi-gigabyte files.

Why does WebRTC need a signaling server if it is peer-to-peer?

WebRTC needs a signaling server only to help the two peers discover each other's network addresses and negotiate connection parameters. Once the direct connection is established, the signaling server is no longer involved in the data transfer.

What happens if one peer goes offline during a WebRTC transfer?

The transfer will pause. If both peers come back online, the transfer can resume from where it left off using resumption protocols built on top of WebRTC data channels.
